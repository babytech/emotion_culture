# 第五阶段开发计划（身份收口 + 稳定性增强 + 轻增长能力）

## 阶段定位

第五阶段定义为“身份收口 + 主链路稳定性增强 + 轻增长能力补齐阶段”。

本阶段承接第四阶段封板之后的小程序产品化升级，重点不是继续做壳层重构，而是解决以下三类问题：

- 用户身份与授权入口不清晰
- 5G / 弱网下主分析链路仍有超时与卡顿问题
- 分享、内容回看、视觉质感等增长体验仍不足

本阶段将正式进入“可持续运营”的产品化建设，而不是继续停留在前端重构阶段。

## 与第四阶段的边界

第四阶段封板范围仍然固定为：

- 小程序主壳层、首页、记录页、分析页、收藏页、我的页、结果页的收口
- 新旧页面信息架构统一
- 结果页和分析页真机收尾
- 第四阶段实施清单、QA 回归、验收报告产出

第五阶段才纳入以下事项：

- 首次进入授权门与身份收口
- 主分析链路超时与卡顿优化
- 自拍与录音输入治理
- 分享到朋友圈能力补齐
- “历史上的今天”内容模块
- 整体视觉风格再升级

## 阶段目标

第五阶段需要达成以下目标：

1. 用户第一次使用小程序时，身份关系、授权状态、后续历史/积分绑定关系清晰
2. 点击“开始分析”后，在 5G / 弱网等真实网络场景下，结果打开成功率明显提升
3. 自拍、录音等输入能力的授权与提交流程更可控，减少无效输入
4. 分享和回看能力从“可用”升级为“可传播、可复访”
5. 小程序视觉质感进一步提升，更接近成熟消费级小程序体验

## 当前输入问题与第五阶段纳入项

### 1. 首次进入授权门与微信身份绑定

#### 当前状态

当前生产环境已经默认依赖微信云托管注入的微信自然身份作为后端主身份，策略为：

- 有 `unionid` 时优先用 `unionid`
- 无 `unionid` 时回落到 `openid`
- 历史记录、留存、积分、后续会员权益统一绑定到这个微信身份主键

相关实现见：

- [config/index.js](/Users/babytech/github/emotion_culture/apps/wechat-mini/config/index.js)
- [miniprogram-api.md](/Users/babytech/github/emotion_culture/services/wechat-api/miniprogram-api.md)
- [user_identity.py](/Users/babytech/github/emotion_culture/services/wechat-api/app/core/user_identity.py)

也就是说，从后端绑定历史记录、留存、配额、积分的角度，当前已经具备“按微信身份绑定”的基础；第五阶段要做的是把这层关系在前端登录门和产品表达上彻底说清楚。

#### 当前问题

虽然底层身份已经有了，但前端缺少明确的“首次进入授权门”：

- 用户打开首页时没有先看到身份确认与授权说明
- 用户难以感知后续历史、会员、积分等会绑定到当前微信身份
- 产品层缺少统一的隐私说明与进入门槛

#### 第五阶段方案

- 新增“小程序首次进入授权页 / 进入门”
- 用户在进入首页前，先完成：
  - 使用说明
  - 隐私与数据使用说明
  - 微信身份授权确认
- 授权成功后再进入首页
- 后续历史、积分、会员、留存均统一绑定到当前微信身份主键（`unionid` 优先，`openid` 兜底）

#### 设计边界

- 这里要区分“微信身份绑定”和“头像昵称资料授权”
- 第五阶段必须保证的是：进入首页前完成身份确认与授权门
- 头像、昵称等资料型授权是否和首次进入强绑定，需要按平台规则与审核风险进一步收敛，不应写成硬依赖

## 2. 主分析链路在 5G / 弱网场景下仍有卡顿与超时

#### 当前问题

当前分析页已经有异步任务与轮询恢复机制，但从真实体验上看：

- 点击“开始分析”后仍会出现“云端超时”
- 用户感知上仍是“结果页出不来”
- 即使服务端异步任务还在运行，前端恢复体验也不够自然

相关实现见 [analyze/index.js](/Users/babytech/github/emotion_culture/apps/wechat-mini/pages/analyze/index.js)。

#### 第五阶段方案

- 将“点击开始分析 -> 结果页打开”定义为第五阶段最高优先级指标
- 重点优化以下链路：
  - 上传阶段耗时
  - 异步任务创建成功率
  - 任务轮询稳定性
  - 结果页自动恢复体验
  - 超时后的用户可见反馈
- 引入更细的链路观测：
  - 上传耗时
  - 创建任务耗时
  - 排队时长
  - 轮询完成率
  - 结果页打开成功率

#### 建议做法

- 优先保留异步主链路，不回退成同步直等
- 进入分析页后，如存在待完成任务，自动恢复查询
- 对图片、音频上传继续做限流与压缩优化
- 在服务端继续查找真正的超时热点，而不是只在前端层面改文案

## 3. 自拍入口需显式摄像头授权

#### 当前问题

当前自拍流程直接调用 `wx.chooseMedia` / `wx.chooseImage` 打开相机，代码中没有独立的 `scope.camera` 授权前置检查，见 [analyze/index.js](/Users/babytech/github/emotion_culture/apps/wechat-mini/pages/analyze/index.js)。

这导致：

- 自拍入口授权意图不够明确
- 若用户拒绝授权，缺少统一的拒绝后处理路径
- 和录音入口相比，摄像头权限反馈不够“显式”

#### 第五阶段方案

- 自拍前先显式检查摄像头权限
- 若未授权，则先弹出权限说明
- 用户拒绝授权时，不进入拍摄流程，直接返回并提示去设置中开启
- 自拍入口和录音入口统一权限反馈风格

## 4. 录音未结束时直接提交，需要显式阻断或自动收口

#### 当前问题

当前提交逻辑没有先拦截“正在录音中”的状态。若用户还在录音就点击提交，当前实现可能只带文字 / 自拍进入分析，最终出现：

- 结果页没有录音识别结果
- 邮件里也没有录音附件
- 用户主观上会觉得“我明明录了，为什么没带上”

相关状态见：

- [analyze/index.wxml](/Users/babytech/github/emotion_culture/apps/wechat-mini/pages/analyze/index.wxml)
- [analyze/index.js](/Users/babytech/github/emotion_culture/apps/wechat-mini/pages/analyze/index.js)

#### 第五阶段方案

推荐采用“自动收口优先，失败再阻断”的策略：

- 若用户录音中点击提交，系统先尝试自动停止录音
- 停止成功后，再继续走录音校验与上传
- 若停止失败，明确提示用户“请先结束录音再提交”

#### 这样做的原因

- 比纯拒绝更顺滑
- 比静默忽略录音更符合用户预期
- 能最大程度减少“我以为录上了，其实没带上”的误解

## 5. 从结果页返回继续分析时，是否保留上一次输入

#### 当前状态

结果页点击“再次分析”后是 `switchTab` 回到分析页，见 [result.js](/Users/babytech/github/emotion_culture/apps/wechat-mini/pages/result/result.js)。

由于分析页是 tab 页面，当前上一次的：

- 文字
- 自拍图片
- 录音文件

都会保留。

#### 第五阶段判断

默认应清空，不建议继续保留全部输入。

#### 原因

- 用户点击“再次分析”，语义上更接近“开始新一次分析”
- 保留上一次自拍和录音，容易造成误用
- 保留旧输入会让用户误以为系统已经重置，但实际上没有

#### 第五阶段方案

- 点击“再次分析”默认清空：
  - 文本
  - 自拍
  - 录音
  - 提交状态
- 如果后续确实需要保留编辑能力，应单独提供一个次级入口，例如：
  - `基于上次内容继续编辑`

而不应把“再次分析”做成默认复用旧输入。

## 6. 增添分享情绪卡片到朋友圈能力

#### 当前状态

当前已经有分享卡页与 `onShareAppMessage`，见 [share/index.js](/Users/babytech/github/emotion_culture/apps/wechat-mini/pages/share/index.js)。

但还没有朋友圈分享链路。

#### 第五阶段方案

第五阶段纳入“朋友圈分享链路补齐”，并明确收敛为：

- 支持当前分享页分享到朋友圈
- 支持生成适合朋友圈传播的静态分享卡
- 支持在分享页中明确引导用户分享到朋友圈

#### 平台边界

- 不应写成“小程序内任意按钮一键直发朋友圈”
- 更可落地的方案是：
  - 当前分享页启用 `onShareTimeline`
  - 或生成分享图并引导用户发朋友圈

#### 交付建议

- 朋友圈分享文案以“轻内容、轻表达”为主
- 朋友圈卡面不直接暴露过多隐私输入
- 分享页需要专门适配朋友圈单页场景

## 7. “历史上的今天”功能

#### 第五阶段方案

该功能可以纳入第五阶段，建议形态如下：

- 放在首页或记录页
- 采用“展开 / 收拢”形式
- 默认收拢，不打断主任务
- 内容定位为“情绪陪伴型知识补充”，不是主分析结果
- 用户在提交今天情绪后，可顺带看到“历史上的今天发生过什么”

#### 数据来源建议

建议采用“成熟 AI 模型搜索与整理”的路线，但必须由自有后端统一承接，而不是让前端直接拼接搜索结果。

当前更稳妥的实现路线是：

- 方案 A：由后端调用成熟 AI 模型，基于日期执行联网搜索 / 检索，再生成简洁摘要
- 方案 B：AI 搜索结果与结构化历史数据源结合，由后端统一整理输出
- 方案 C：将高频日期结果做缓存，避免每次都实时搜索

#### 原因

- 该内容适合借助成熟 AI 模型提升可读性和趣味性
- 但搜索结果必须先经过后端层做事实收敛、长度控制、内容审核与缓存
- 不应把 AI 原始搜索回答直接暴露给用户
- 第五阶段更适合把它做成“AI 检索增强的可控内容模块”

#### 额外边界

- 必须区分“历史事实”和“情绪建议”
- 历史内容不应过长
- 需要审核极端事件、灾难事件、政治事件等高敏感内容
- 输出建议固定为：
  - 1 条主事件摘要
  - 1 段不超过约 80~120 字的简述
  - 可选 1 句轻量延伸文案
- 若当天搜索结果不稳定或不适合展示，应允许直接降级为“不展示该模块”

## 8. UI 视觉需要从“可用”升级到“成熟消费级”

#### 当前判断

这个反馈是成立的。

第四阶段已经完成了壳层统一，但当前视觉语言仍更偏“稳妥可用”，距离成熟消费级小程序常见的“记忆点、节奏感、品牌感”还有差距。

#### 第五阶段视觉方向建议

- 统一主视觉关键词：
  - 东方感
  - 轻情绪
  - 少年感
  - 现代国风而不是传统模板风
- 提升以下设计层：
  - 更有辨识度的首页 Hero
  - 更高级的卡片材质与层次
  - 更统一的按钮体系与状态动画
  - 更有记忆点的配色，不只停留在“暖米色 + 红色点缀”
- 避免的问题：
  - 过多说明性文字
  - 卡片堆叠但缺少重点
  - 所有页面长得太像，缺乏页面角色差异

#### 推荐改造顺序

1. 首页
2. 结果页
3. 分享页
4. 记录页 / 我的页

## 第五阶段里程碑建议

### M1：身份与授权门

- 首次进入授权页上线
- 微信身份绑定链路统一
- 摄像头权限前置校验上线
- 录音中提交流程收口上线

### M2：主链路稳定性专项

- 上传 / 异步任务 / 轮询观测补齐
- 5G / 弱网超时问题专项治理
- 自动恢复待完成分析任务
- 再次分析默认清空输入

### M3：分享与内容增强

- 分享页支持当前页分享到朋友圈链路
- 情绪卡文案与卡面升级
- “历史上的今天”AI 搜索折叠模块上线

### M4：视觉升级收口

- 首页视觉二次升级
- 结果页视觉与动作区二次升级
- 分享页传播感升级
- 全局动效与状态一致性收口

## 第五阶段暂不纳入事项

- 重会员商城
- 大规模商业化组件
- 长语音深度转写
- 多轮对话式情绪分析
- 复杂社交互动体系

## 当前建议结论

第五阶段应聚焦在“身份明确、链路更稳、传播更强、视觉更成熟”这四件事上。

其中优先级排序建议为：

1. 首次授权门 + 微信身份收口
2. 5G / 弱网下分析链路稳定性专项
3. 自拍 / 录音输入治理
4. 朋友圈分享与“历史上的今天”
5. UI 二次升级
