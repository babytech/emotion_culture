# 第二阶段开发计划（方案 B）

## 文档目的

本文件用于将 [第二阶段实施清单](/Users/babytech/github/emotion_culture/docs/stage1/phase2-implementation-checklist.md) 进一步拆解为可执行的开发顺序、依赖关系、阶段里程碑与验收方式。

本计划遵循 [产品共识文档](/Users/babytech/github/emotion_culture/docs/product-consensus.md) 中“第二阶段：用户留存与粘性方向”的已达成共识。

配套任务级拆分见：[phase2-task-breakdown.md](/Users/babytech/github/emotion_culture/docs/stage1/phase2-task-breakdown.md)。

## 第二阶段目标

在第二阶段结束时，项目应具备以下能力：

- 小程序形成可持续使用的留存闭环：日历、打卡、周报、收藏
- 后端提供可复用的留存聚合能力与稳定 API
- `PC` 提供轻量留存回看入口，保持与小程序数据口径一致
- 在不引入重会员体系前提下，提升用户连续使用与复盘频率
- 保持第一阶段隐私边界，不扩大原始媒体长期留存范围

## 方案选择

### 方案 A（数据洞察优先）

- 重点：情绪日历、趋势图、周报/月报深度洞察
- 特点：分析价值强、增长激励弱

### 方案 B（平衡留存优先）

- 重点：日历 + 连续打卡 + 周报 + 收藏
- 特点：留存闭环完整、实现复杂度适中、可在现有架构快速落地

### 方案 C（增值玩法优先）

- 重点：积分/虚拟币、兑换、AI 海报/画像等奖励能力
- 特点：增长刺激强，但成本与风控复杂度高

### 当前执行选择

第二阶段采用 **方案 B**。

- 本阶段不把 `SPEECH_STT_ENDPOINT` 作为 ASR 开关
- `SPEECH_ASR_SERVICE` 作为 ASR 开关，`VOICE_REQUIRE_TRANSCRIPT` 仅代表严格度
- 方案 A/C 中未覆盖能力在下一阶段按任务包补齐

## 开发策略

### 总体原则

- 复用第一阶段稳定链路，新增能力尽量做“旁路扩展”
- 先做后端聚合与模型，再做端侧页面承接
- 先做小程序主留存阵地，再补 `PC` 轻量回看
- 先做可验证的留存闭环，再考虑复杂奖励与商业化

### 交付顺序

第二阶段建议按以下顺序推进：

1. 固定留存数据模型与聚合接口
2. 打通小程序日历与打卡主流程
3. 打通周报与收藏链路
4. 补齐 `PC` 轻量回看能力
5. 补齐数据删除与留存策略
6. 完成全量回归与验收封板

## 阶段拆分

### 阶段 A：留存数据底座

目标：

- 固定打卡、日历、周报、收藏的数据结构
- 固定后端聚合口径，避免端内重复计算

主要任务：

- 设计留存摘要模型与索引
- 增加日历聚合 API
- 增加连续打卡计算服务
- 增加周报聚合 API
- 增加收藏读写 API

阶段完成标志：

- 小程序可依赖统一后端口径渲染留存能力
- 主要接口具备可回归的稳定输入输出

### 阶段 B：小程序留存闭环

目标：

- 形成“查看日历 -> 完成打卡 -> 回看周报 -> 收藏内容”的闭环

主要任务：

- 新增情绪日历页
- 新增打卡状态与连续天数展示
- 新增周报入口与周摘要卡片
- 结果页增加收藏能力（诗词/角色）
- 增加分享卡片导出（非 AI）

阶段完成标志：

- 用户可在小程序独立完成一周内的持续记录与回看

### 阶段 C：PC 轻量留存补齐

目标：

- `PC` 提供留存能力的只读或轻编辑承接，不与小程序分叉

主要任务：

- 新增近期情绪趋势摘要
- 新增最近周报回看入口
- 新增收藏内容轻量回看

阶段完成标志：

- `PC` 用户可回看近期变化与收藏，不影响主分析流程

### 阶段 D：数据策略与稳定性回归

目标：

- 确保新增留存能力满足隐私策略、性能目标与可恢复标准

主要任务：

- 删除、清空、关闭留存开关回归
- 聚合接口耗时与失败降级回归
- 小程序与 `PC` 数据口径一致性回归

阶段完成标志：

- 主留存场景可完成
- 异常可恢复
- 数据口径可验证

## 依赖关系

### 核心依赖

- 阶段 A 完成后，阶段 B 与 C 才能稳定展开
- 阶段 B 完成后，再做阶段 D 的端到端回归最有效
- 阶段 A 的接口口径冻结后，`PC` 适配成本最低

### 不建议并行的部分

- 不建议在留存模型未定前并行做周报页面细化
- 不建议在收藏能力未定前并行做复杂分享模板
- 不建议在回归未通过前提前引入积分与 AI 奖励链路

## 开发优先级

### P0

- 留存数据模型
- 日历聚合 API
- 连续打卡服务
- 小程序日历页
- 小程序打卡入口

### P1

- 周报聚合 API
- 小程序周报页
- 收藏 API 与前端入口
- `PC` 留存摘要回看

### P2

- 分享卡片细节优化
- 周报视觉优化
- 留存文案与引导优化

## 验收里程碑

### 里程碑 1：后端留存底座完成

验收标准：

- 打卡、日历、周报、收藏 API 可用
- 聚合口径固定并完成接口自测

### 里程碑 2：小程序留存闭环完成

验收标准：

- 日历可看
- 打卡可做
- 周报可回看
- 收藏可读写

### 里程碑 3：PC 轻量承接完成

验收标准：

- `PC` 可回看近期趋势与周报摘要
- 收藏可在 `PC` 侧查看

### 里程碑 4：第二阶段封板

验收标准：

- 全量回归通过
- 清单完成率达到封板标准
- 验收报告与回归报告更新

## 风险与控制点

### 数据口径风险

- 风险：日历、周报、趋势在不同端出现计算分叉
- 控制：统一后端聚合，端内仅渲染

### 留存体验风险

- 风险：新增页面多但用户感知收益不明显
- 控制：每个页面至少提供一个可执行结论或下一步动作

### 范围失控风险

- 风险：第二阶段混入积分商城和 AI 奖励导致延期
- 控制：明确把方案 C 相关能力移到下一阶段独立推进

## 与下一阶段衔接（A/C 未覆盖项）

方案 B 未覆盖的方案 A/C 内容，在下一阶段按以下包落地：

- A 包：月报、深度趋势、多维筛选、异常波动解释
- C 包：积分/虚拟币账本、兑换规则、AI 奖励内容生成与风控
