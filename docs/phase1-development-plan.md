# 第一阶段开发计划

## 文档目的

本文件用于将 [第一阶段实施清单](/Users/babytech/github/emotion_culture/docs/phase1-implementation-checklist.md) 进一步拆解为可执行的开发顺序、依赖关系、阶段里程碑与验收方式。

本计划严格遵循 [产品共识文档](/Users/babytech/github/emotion_culture/docs/product-consensus.md) 中已经封板的第一阶段范围。

## 第一阶段目标

在第一阶段结束时，项目应具备以下能力：

- 小程序可稳定完成文字、语音、自拍三类输入分析
- `PC` 可稳定完成文字、录音、摄像头拍照三类输入分析
- 双端共用统一分析核心，结果口径基本一致
- 结果页、邮件导出、简版历史记录页、极简设置页可用
- 历史记录、隐私边界、删除能力和基础状态反馈可用

## 开发策略

### 总体原则

- 先打通核心主链路，再补页面与附属能力
- 先解决后端统一分析与数据结构，再做端侧适配
- 先做小程序主阵地，再补 `PC` 配套能力
- 先保证稳定与可恢复，再考虑体验精修

### 交付顺序

第一阶段建议按以下顺序推进：

1. 统一后端分析与数据结构
2. 打通小程序主分析链路
3. 补齐语音与自拍边界处理
4. 完成邮件导出能力
5. 完成历史记录底座与简版历史记录页
6. 完成极简设置页与隐私说明
7. 补齐 `PC` 摄像头拍照与录音链路
8. 做稳定性回归与双端口径校准

## 阶段拆分

### 阶段 A：后端统一与主链路收口

目标：

- 固定第一阶段结果结构
- 固定输入模式定义
- 固定历史记录摘要结构
- 明确语音与自拍的后端处理边界

主要任务：

- 统一分析接口输入结构
- 统一分析接口输出结构
- 定义输入模式枚举，例如文字、语音、自拍、摄像头拍照
- 固定结果卡片字段
- 固定历史记录摘要字段
- 增加语音质量校验与重录提示返回
- 增加人脸可用性校验与重拍提示返回

阶段完成标志：

- 后端可以稳定返回统一格式结果
- 小程序与 `PC` 均可基于统一结果结构开发

### 阶段 B：小程序主流程打通

目标：

- 小程序完成第一阶段最小可用产品闭环

主要任务：

- 保留文字输入
- 保留录音输入
- 新增或完善前置自拍流程
- 去除普通相册图片分析入口
- 打通文字分析
- 打通语音转文字后分析
- 打通自拍拍照分析
- 结果页按固定结构渲染
- 增加上传中、分析中、结果生成中状态反馈
- 支持失败重试与输入保留

阶段完成标志：

- 小程序可以完成一次完整的文字、语音、自拍分析闭环

### 阶段 C：邮件导出与可恢复体验

目标：

- 邮件成为可选导出能力，而不阻塞主流程

主要任务：

- 结果页提供邮箱输入与校验
- 邮件接口与分析接口解耦
- 明确邮件发送状态
- 支持邮件失败后重试
- 如本次使用自拍，支持将本次自拍图纳入邮件
- 保证邮件失败不影响结果查看

阶段完成标志：

- 用户可在分析后选择发送结果邮件
- 邮件失败不会破坏主流程

### 阶段 D：历史记录与设置页

目标：

- 形成第一阶段的“可回看”和“可控制”基础能力

主要任务：

- 存储轻量历史摘要
- 提供小程序简版历史记录页
- 支持单条删除与清空全部
- 提供极简设置页
- 提供“保存我的历史记录”开关，默认开启
- 展示 24 小时原始媒体保留说明
- 展示 180 天摘要保留说明
- 展示非医疗诊断说明
- 提供问题反馈入口

阶段完成标志：

- 用户可以回看最近记录
- 用户可以控制历史保存与删除

### 阶段 E：PC 补齐与双端收口

目标：

- `PC` 端补齐第一阶段最小能力，与小程序保持统一逻辑

主要任务：

- 保留文字输入
- 保留录音输入
- 新增或完善摄像头拍照入口
- 拍照后预览与确认
- 去除普通本地图片情绪分析入口
- 增加拍照失败恢复能力
- 接入统一分析核心
- 增加轻量本地历史能力

阶段完成标志：

- `PC` 可完成文字、录音、摄像头拍照三类输入分析
- 与小程序主情绪判断和解释方向基本一致

### 阶段 F：稳定性回归与封板

目标：

- 满足第一阶段最低稳定性标准

主要任务：

- 文本分析性能回归
- 语音链路失败回退回归
- 自拍与摄像头链路回归
- 邮件发送回归
- 历史记录写入与删除回归
- 设置开关回归
- 双端结果口径对比
- 文案与隐私说明核对

阶段完成标志：

- 主流程能完成
- 异常能降级
- 等待有反馈
- 失败可恢复
- 双端口径一致

## 依赖关系

### 核心依赖

- 阶段 A 完成后，阶段 B、C、D、E 才能稳定展开
- 阶段 B 完成后，阶段 C 与 D 的小程序页面开发效率最高
- 阶段 A 与 B 形成稳定结果结构后，再做 `PC` 端适配，避免重复返工

### 不建议并行的部分

- 不建议在结果结构未定前并行开发历史记录页
- 不建议在语音边界未定前做复杂语音体验
- 不建议在小程序主链路未稳前投入第二阶段留存功能

## 开发优先级

### P0

- 统一分析结构
- 小程序文字分析
- 小程序自拍分析
- 语音转文字主链路
- 结果页固定结构
- 邮件不阻塞主流程

### P1

- 小程序历史记录页
- 小程序设置页
- 历史保存开关
- 删除能力
- `PC` 摄像头拍照分析
- `PC` 录音分析

### P2

- `PC` 本地轻量历史能力
- 体验细节优化
- 文案细化与异常提示优化

## 验收里程碑

### 里程碑 1：后端统一完成

验收标准：

- 统一结果结构已固定
- 统一输入模式已固定
- 语音与自拍边界返回明确错误提示

### 里程碑 2：小程序主流程完成

验收标准：

- 文字可分析
- 语音可分析
- 自拍可分析
- 结果页可完整展示

### 里程碑 3：留存与导出基础完成

验收标准：

- 邮件导出可用
- 简版历史记录页可用
- 设置页可用
- 删除与保存开关可用

### 里程碑 4：PC 第一阶段完成

验收标准：

- `PC` 文字、录音、摄像头拍照分析可用
- `PC` 与小程序口径基本一致

### 里程碑 5：第一阶段封板

验收标准：

- 满足第一阶段最低稳定性标准
- 通过一次完整双端回归
- 文档与实际实现一致

## 风险与控制点

### 语音风险

- 风险：杂音、识别差导致误判
- 控制：先做语音质量校验，不合格直接拒绝处理

### 自拍风险

- 风险：人脸不可用、多人脸、模糊或遮挡导致误判
- 控制：先做人脸可用性校验，不合格直接重拍

### 邮件风险

- 风险：发送慢或失败拖垮主流程
- 控制：邮件独立处理，不阻塞结果展示

### 历史记录风险

- 风险：隐私边界不清晰
- 控制：只存摘要、原始媒体 24 小时自动删除、允许用户随时删除

### 范围失控风险

- 风险：第一阶段不断混入第二阶段功能
- 控制：严格以 [产品共识文档](/Users/babytech/github/emotion_culture/docs/product-consensus.md) 的封板范围为准
